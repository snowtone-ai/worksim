# Cβ Role Differentiation Guardrails

## Why This Exists

Cβはナラティブゲームとして「選択が後に影響する」体験を強めた。これは没入感に効く一方、どの職種にもある顧客・上司・他部署・締切の調整へ収束しやすい。

品質ゲートが多すぎる場合も、全シナリオが同じ骨格に寄りやすい。チェック項目を増やすだけではなく、各職種の仕事材料と判断単位を先に固定する必要がある。

## Balance

ナラティブ性は捨てない。ステークホルダーも消さない。ただし、主役は調整ではなく、職種固有の仕事材料を使った判断にする。

顧客、上司、他部署、締切は、職種固有の判断を浮かび上がらせる圧力装置として扱う。

## Role Work Kernel

`roleWorkKernel` は「この職種は、何を何に変換する仕事か？」を構造化objectで定義する。旧文字列はloader互換のため残すが、Cβ canonical dataでは使わない。

Required fields:

- input
- output
- transformation
- constraints
- workArtifacts
- metrics
- failureModes
- evaluationCriteria
- nonGenericReason

- 地方銀行 法人営業/融資担当: 顧客の資金不安を、返済原資・保全・保証・条件・記録を備えた融資案に変換する仕事。
- 商品企画: 顧客の不満と市場機会を、原価・製造・品質保証・営業・ブランドに通る発売仕様と発売条件に変換する仕事。

## Anti-Generic Stakeholder Coordination Rule

WorkSimのCβシナリオでは、ステークホルダー調整そのものを職業体験の主役にしてはならない。

各シーンの主役は、必ずその職種固有の仕事材料・数字・成果物・判断基準・失敗リスクであること。

職種名だけを置換して成立する「汎用調整シーン」はFailとする。

## Role Specificity Checks

- その職種固有の資料・成果物・数値が最低2つ以上ある
- その職種固有の失敗リスクがある
- その職種固有の評価基準がある
- 選択肢が単なる「寄り添う / バランスを取る / 厳しくする」になっていない
- ステークホルダー会話だけで判断が成立する場合はFail
- 職種名を別職種に置換しても成立する場合はFail
- 仕事材料を外しても成立するシーンはFail

## Decision Primitive Diversity

Cβでは、各sceneに `coreDecisionPrimitive` を持たせる。

金融:

- 資金不足の初期切り分け
- 返済原資評価
- 保証条件設計
- 稟議説明補強
- 条件付き最終回答

商品企画:

- 顧客不満の仕様化
- 原価制約下の仕様削減
- 根拠ある売り文句設計
- 品質表示リスク判定
- 初回数量と発売条件判断

## Comparison

地方銀行 法人営業/融資担当は、資金繰り表、売掛入金、保証協会条件、稟議コメントを使う。良い判断は、顧客安心と銀行の説明責任を両立する融資条件へ翻訳できること。

商品企画は、顧客アンケート、競合比較、原価表、品質保証コメント、発売判定シートを使う。良い判断は、顧客価値を発売可能な仕様と発売条件へ翻訳できること。

## Checklist For New Roles

- roleWorkKernelを1文で書いたか
- 5 scenes / 5 tasksになっているか
- 各sceneにworkMaterialがあるか
- 各sceneにcoreDecisionPrimitiveがあるか
- 各sceneにkernelConnectionがあるか
- 各sceneに職種固有の資料・数字・失敗リスク・評価基準があるか
- ステークホルダー調整だけで判断が成立していないか
- 職種名だけ置換しても成立しないか
- 仕事材料を外すと選択できない状態になっているか
