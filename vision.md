# WorkSim — vision.md

## 1. プロダクト概要

**プロダクト名**：WorkSim（ワークシム）

**一言で**：日本の就活生のための、職場の1日を体験できる Web シミュレーションゲーム

**解決する課題**：説明会・適職診断では伝わらない「実際の業務イメージ」を就職前に提供し、入社後ミスマッチを減らす

**ターゲットユーザー**：
- プライマリ：日本の就活前〜就活中の大学生（特に国立大学）
- セカンダリ：学生にリーチしたい企業の人事
- 営業ターゲット：国立大学のキャリアセンター（最初は大分大学）

## 2. ビジネスモデル

| フェーズ | ユーザー | 課金 |
|---------|---------|------|
| MVP | 学生 | 無料 |
| MVP | 大分大学 | 無料（実証実験） |
| 拡大期 | 全国の大学 | 要相談 |
| 拡大期 | 企業 | カスタムシナリオ作成機能で有料 |

## 3. MVP スコープ

### 3.1 含むもの

1. **業界選択**：IT業界のみ（MVP）
2. **職種選択**：6職種（Webエンジニア / ITコンサル / 法人営業 / PdM / インフラ運用 / UIデザイナー）
3. **シミュレーション体験**：1職種あたり 20分（後で調整可）
4. **Google ログイン**：Supabase Auth 経由
5. **任意プロフィール入力**：大学名・学部・志望業界（個人名は取らない）
6. **適性スコア＋特性タグ**：体験後に表示
7. **SNSシェア機能**：体験結果のスクリーンショット共有

### 3.2 含まないもの（MVP外、案だけ用意）

- IT以外の業界
- 企業向けカスタムシナリオエディタ
- 結果分岐（同じ選択でも結果が変わる動的シナリオ）
- NPC との自由会話（LLM応答）
- 音声・BGM
- スマホ対応
- 多言語対応

### 3.3 開発戦略：1職種完全版 → 残職種展開

**なぜこの順序か**：6職種を並行して薄く作ると、シナリオ品質が職種ごとにばらつき、修正不能になる（バイブコーディング失敗の典型パターン）。1職種を完璧に仕上げて、シナリオ生成からタスクUI実装までの **再現性のあるワークフロー** を確立してから、残り5職種に同じワークフローを適用する。

**完成定義（1職種目＝Webエンジニア）**：
- シナリオ JSON が完成
- 全タスク UI が動作
- 適性スコア計算が正しい
- 体験完了から結果表示まで 1 度もエラーなくプレイ可能
- リサーチ手順 → JSON 生成 → 実装 が `scenarios/_research-plan.md` に文書化されている

## 4. ユーザーストーリー（Given / When / Then）

### US-1：業界・職種選択
- **Given**：未ログインの学生がトップページにアクセスした
- **When**：「IT業界」を選択し、表示された6職種から「Webエンジニア」を選んだ
- **Then**：体験開始前のサマリー画面（所要20分・新卒3年目・概要）が表示され、「Googleでログインして開始」ボタンが表示される

### US-2：Google ログイン
- **Given**：体験開始前サマリー画面で「Googleでログイン」を押した
- **When**：Google OAuth 画面で承認した
- **Then**：プロフィール任意入力画面（大学名・学部・志望業界・全部スキップ可）が表示される

### US-3：プロフィール（任意）
- **Given**：プロフィール入力画面が表示された
- **When**：何も入力せず「スキップ」を押した
- **Then**：シミュレーションが開始される（プレイには支障なし）

### US-4：シミュレーションプレイ
- **Given**：Webエンジニアのシミュレーションが開始された
- **When**：オフィス背景でデスクのオブジェクトをクリックした
- **Then**：朝のメールチェックタスクが起動し、メールUI上で実際にメールを読んで返信できる

### US-5：選択肢による分岐（適性スコアのみ）
- **Given**：ペアプロのタスクで「先輩に質問する」「自分で調べる」の選択肢が出た
- **When**：「自分で調べる」を選んだ
- **Then**：シナリオは進むが、内部スコアに `engineer: +5, traits: ["自己解決型"]` が加算される（**シナリオ自体は分岐しない、MVP制約**）

### US-6：結果表示
- **Given**：全タスクを完了した
- **When**：「終了」ボタンを押した
- **Then**：結果画面が表示され、職種別適性スコア（Webエンジニア 78点）と特性タグ（「コツコツ型」「自己解決型」）と SNSシェアボタンが表示される

### US-7：リプレイ
- **Given**：1度プレイ完了したユーザーが再度同じ職種を選んだ
- **When**：「もう一度プレイ」を押した
- **Then**：別の選択肢を選び直して、別のスコアを得られる（過去のプレイ履歴は別レコードとして保存）

### US-8：別職種挑戦
- **Given**：Webエンジニアをプレイ済みのユーザーが結果画面にいる
- **When**：「ITコンサルも体験する」を押した
- **Then**：ITコンサルのシミュレーションが開始される。完了後、複数職種スコアの比較は表示しない（MVP外）

### US-9：エラー時
- **Given**：プレイ中に通信エラーが発生した
- **When**：再接続できた
- **Then**：直前のタスクから再開できる（プレイ進捗は localStorage に自動保存）

### US-10：デプロイ確認
- **Given**：Claude Code が `pnpm build` を成功させた
- **When**：`vercel --prod` を実行した
- **Then**：本番URLでログイン→プレイ→結果表示までエラーなく動作する

## 5. シナリオデータ構造

```
scenarios/
├── _schema.md              # スキーマ仕様
├── _research-plan.md       # リサーチ計画
├── it/
│   ├── web-engineer.json   # 1職種目（先行）
│   ├── consultant.json
│   ├── b2b-sales.json
│   ├── pdm.json
│   ├── infra-engineer.json
│   └── ui-designer.json
└── _template.json          # 新規シナリオの雛形
```

シナリオ JSON の構造：
```
シナリオ
  ├─ メタ情報（業界、職種、所要時間、出典URL一覧）
  ├─ タスク[] （時系列順）
  │    ├─ シーン背景（office-desk / meeting-room / outdoor 等）
  │    ├─ 作業案内テキスト
  │    ├─ NPC台詞（任意）
  │    ├─ 選択肢[]
  │    │    └─ 選択肢ごとに { aptitude_delta, traits[] }
  │    └─ 完了条件
  └─ スコア計算ルール
```

詳細は `scenarios/_schema.md`。

## 6. データモデル（Supabase）

### auth.users（Supabase 標準）
- Google が提供する個人情報（名前・メール）はここに保存される
- アプリは参照しない（個人情報の最小化）

### public.profiles
| カラム | 型 | 必須 | 備考 |
|-------|----|------|------|
| user_id | UUID | ✅ | auth.users への FK、PK |
| university | TEXT | ❌ | 任意入力（例: 大分大学） |
| faculty | TEXT | ❌ | 任意入力（例: 経済学部） |
| target_industry | TEXT | ❌ | 任意入力（例: IT） |
| created_at | TIMESTAMPTZ | ✅ | デフォルト now() |
| updated_at | TIMESTAMPTZ | ✅ | デフォルト now() |

**RLS ポリシー**：
- SELECT：自分のレコードのみ
- INSERT/UPDATE：自分のレコードのみ

### public.play_sessions
| カラム | 型 | 必須 | 備考 |
|-------|----|------|------|
| id | UUID | ✅ | PK、デフォルト gen_random_uuid() |
| user_id | UUID | ✅ | auth.users への FK |
| scenario_id | TEXT | ✅ | 例: "it/web-engineer" |
| started_at | TIMESTAMPTZ | ✅ | |
| completed_at | TIMESTAMPTZ | ❌ | 完了時のみ |
| choices | JSONB | ✅ | 選択肢ID配列 |
| aptitude_score | JSONB | ❌ | 完了時のみ {"engineer": 78} 形式 |
| traits | TEXT[] | ❌ | 完了時のみ ["コツコツ型", "自己解決型"] |

**RLS ポリシー**：
- SELECT：自分のレコードのみ
- INSERT/UPDATE：自分のレコードのみ

### public.scenarios（読み取り専用、JSON ファイルから seed）
| カラム | 型 | 必須 |
|-------|----|------|
| id | TEXT | ✅ |
| industry | TEXT | ✅ |
| role | TEXT | ✅ |
| duration_minutes | INTEGER | ✅ |
| data | JSONB | ✅ |
| version | INTEGER | ✅ |

**RLS ポリシー**：誰でも SELECT 可能（ログイン不要のシナリオ一覧表示用）

## 8. セキュリティ要件

### MVP 必須
- HTTPS 強制（Vercel デフォルト）
- Supabase RLS 全テーブル設定
- 環境変数（API キー等）は `.env.local` のみ。Git コミット禁止
- profiles テーブルに個人名を保存しない
- Google OAuth のみ（パスワード管理しない）

### 将来強化（大学交渉時に必要なら追加）
- 個人情報削除リクエスト機能（GDPR / 個人情報保護法対応）
- ログイン履歴の閲覧
- 利用規約・プライバシーポリシー（V1.1 で必須）
- データ保持期間の明示
- Cookie 同意バナー

参考：日本の大学が求めるセキュリティ要件のリサーチを Claude Code 側で随時実施し、`docs/security-requirements.md` に追加していく。

### スケール時（同時1000、月1000人プレイ等）
| サービス | プラン | 月額 |
|---------|--------|------|
| Vercel | Pro | $20 |
| Supabase | Pro | $25 |

**月額: 約$45**

### 重要な制約：Supabase Free の自動停止対策

Supabase の Free プロジェクトは **7日無アクティブで自動停止** する。MVP段階では使用頻度が落ちる可能性があるため、対策として：

- GitHub Actions で 5日に1回 Supabase に軽量クエリを送る cron を設定
- これにより停止を回避できる
- 詳細は Phase 2（Supabase セットアップ）で実装

## 11. 成功条件

### MVP 成功基準
- 6職種すべて完成
- 大分大学キャリアセンターにデモ実施
- 100名以上のテストプレイヤーが完走
- 体験プレイ後の「業務イメージが湧いた」回答 70% 以上

### MVP 失敗基準
- シナリオが「リアルさ Level 2」に達しない（実在ツール・専門用語が不正確）
- 体験完了率 50% 未満（途中離脱が多い）
- 個人情報流出インシデント

## 12. リスクと対策

| リスク | 確率 | 影響 | 対策 |
|--------|------|------|------|
| シナリオ品質不足 | 中 | 高 | 1職種目で品質基準を確立、リサーチ手順を文書化 |
| Supabase 自動停止 | 高 | 中 | GitHub Actions keep-alive cron |
| AI生成コードの脆弱性 | 中 | 高 | RLS 強制、CLAUDE.md ルール、テスト先行 |
| 大学交渉でセキュリティ懸念 | 高 | 中 | プロフィールから個人名除外済み、必要なら個人情報削除機能を追加 |
| 1ファイル肥大化 | 中 | 中 | 300行制限、Hook で自動チェック |
| Bypass Permissions による事故 | 低 | 高 | settings.json deny リスト + 危険コマンドブロック |
