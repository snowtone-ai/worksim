# Research Plan — シナリオ生成のための再現性のあるリサーチ手順

このドキュメントは「新しい職種のシナリオを高品質に作るためのワークフロー」を定義する。
1職種目（Webエンジニア）の実装で得た知見をここに追記し、残職種に同じ手順を適用する。

---

## ゴール

「リアルさ Level 2」の達成。

- ✅ 業務フローが正確（タスクの順序・時間配分が実際と一致）
- ✅ 使用ツール・専門用語が実在する（Slack、GitHub、Figma、Salesforce 等）
- ✅ 新卒3年目の典型的な1日として違和感がない
- ❌ 業界経験者の監修（これは Level 3、企業カスタムシナリオ時の責任）

---

## ワークフロー（職種ごとに繰り返す）

### Step 1：一次情報源の収集（30〜60分）

**必須リソース**：

1. **厚生労働省 job tag**（職業情報提供サイト）
   - URL：https://shigoto.mhlw.go.jp/User/
   - 役職名で検索し、「仕事の内容」「労働条件」を抽出
   - 公的データなので信頼性が高い

2. **企業の公式キャリアページ**（最低3社）
   - 代表的なIT企業：リクルート、サイバーエージェント、LINEヤフー、SmartHR、freee 等
   - 「新卒の1日」「若手社員インタビュー」を探す

3. **Wantedly Story / Story 系職種別記事**
   - 「<職種名> 新卒 1日」「<職種名> 業務内容 リアル」で検索
   - 体験談ベースなのでタスクの粒度が分かる

4. **note / Zenn / Qiita（エンジニア系のみ）**
   - 「<職種名> 新卒 入社後 業務」等で検索
   - 注意：個人ブログは主観が強いので、最低2記事から共通項を抽出

5. **業界ニュース**（任意）
   - 業界トレンドを反映させる（例：AI 活用が進む2026年のWebエンジニアは Copilot 利用が標準）

### Step 2：典型的1日の骨格を抽出

以下のテンプレートに沿って情報を整理する：

```
<職種名>の典型的な1日（新卒3年目 / <企業規模>）

08:30〜09:00 出社・準備
09:00〜09:30 [タスクA]
09:30〜10:30 [タスクB]
...
12:00〜13:00 ランチ
...
17:30〜18:00 退勤前準備
```

### Step 3：タスクの「意思決定ポイント」を見つける

各タスクで、新卒3年目が **判断に迷う場面** を抽出する。
これが選択肢のもとになる（適性スコアと特性タグに繋がる）。

例（Webエンジニア）：
- 「バグの調査、どこから始めるか？」
- 「コードレビューで先輩のコードに違和感があるが、指摘するか？」
- 「機能Xの実装、最短経路 vs 拡張性、どちらを選ぶか？」

### Step 4：シナリオ JSON 化（_schema.md 準拠）

`scenarios/_template.json` を雛形にして編集する。

**必須**：`sources[]` フィールドに最低3つの出典 URL を記載する。

**注意**：この手順は公開向け・MVP向けシナリオの標準ルール。企業向けカスタムシナリオでは、契約・許諾・提供素材の範囲で実在名を使用してよいが、利用範囲は JSON と UI に明記する。

### Step 5：ユーザー（souma さん）に確認依頼

JSON 完成後、Claude Code は以下のように要約してユーザーに確認を求める：

```
【シナリオ確認】Webエンジニア職

主要タスク：
1. メールチェック・返信
2. 朝会
3. ペアプログラミング
4. ランチ
5. コードレビュー
6. 仕様検討MTG
7. 実装
8. 日報

選択肢の意思決定ポイント数：12
出典：厚労省 job tag + リクルート キャリアページ + Wantedly 1件

進めてよいですか？修正点があれば指示してください。
```

### Step 6：UI 実装

タスク種別ごとに UI コンポーネントを実装。

`_schema.md` の `ui_component` 一覧にあるものから選び、必要なら新規追加。

### Step 7：プレイテスト

1. 自分（Claude Code）でローカルプレイ → エラー検出
2. ユーザー（souma さん）にプレイ依頼 → リアルさを確認
3. フィードバックを JSON に反映

### Step 8：知見を本ドキュメントに追記

職種完成後、このドキュメントの `## 各職種の知見` セクションに追記：
- どの情報源が役立ったか
- 何で困ったか
- 次の職種で気をつけること

## 各職種の知見

### Webエンジニア

**役立った情報源**

- 厚生労働省 job tag：職務の全体像と業務責任の確認に使う
- IPA DX白書2023：DX文脈、技術者に求められるスキル、チーム開発の背景づけに使う
- 企業公式キャリアページ・若手社員インタビュー：1日の流れ、会議、レビュー、障害対応などの粒度調整に使う

**実装で分かったこと**

- Cβ標準の没入モードはexactly 5シーンの連続体験を基本とし、代表既存シナリオは移行完了まで20シーン構成も許容する。E2Eで中盤以降まで必ず進めて確認する
- 会議シーンは選択肢が長くなりやすく、固定フッターが画面外に押し出される。選択肢部分だけスクロールし、確定ボタンは常に見える構造にする
- Playwright の認証は Google OAuth 本体を使わず、Supabase の `generateLink` + `verifyOtp` でテスト用 Cookie を作る
- Vitest と Playwright の `*.spec.ts` が衝突しないよう、Vitest は `src/**/*.{test,spec}.{ts,tsx}` のみ対象にする

**次の職種で気をつけること**

- まず没入モードのexactly 5シーン連続体験を完成させ、E2Eで結果画面まで通す
- 長文選択肢が出る場面では、PC画面でもボタンが見えるかをPlaywrightで確認する
- シナリオ JSON 内では実在企業名・サービス名・人名を避け、固有名詞は架空名に置き換える

## 共通シーン背景の用意

職種を跨いで使い回せる背景画像を `public/backgrounds/` に配置：

- `office-desk.webp`：自分のデスク
- `meeting-room.webp`：会議室
- `client-office.webp`：クライアント先
- `lounge.webp`：休憩室
- `outdoor.webp`：屋外
- `whiteboard.webp`：ホワイトボード前

**画像の調達**：
- 著作権フリー素材（Unsplash、Pexels）からダウンロード
- AI生成（DALL-E、Stable Diffusion）でも可、ただし商用利用可能なツールを使う
- 1280x720 以上、WebP 形式に変換、500KB 以下に圧縮

## 次の職種を作る手順

1. `docs/state.md` を読み、完了済みタスクと進行中タスクを確認する
2. `docs/decisions.md` を読み、既存の設計判断と矛盾しないことを確認する
3. 対象職種について、厚労省 job tag と企業公式情報を中心に一次情報源を3つ以上集める
4. Cβでは先にRole Differentiation Sheetと構造化 `roleWorkKernel` を作る
5. 典型的な1日の流れをexactly 5シーンに圧縮し、各シーンの意思決定ポイントを1つずつ決める
6. 各sceneで会話より先に `workMaterial` と `roleSpecificity.kernelConnection` を定義する
7. ステークホルダー圧力は許容するが、仕事材料・数字・decision primitiveを主役にする
8. `scenarios/_schema.v2.md` に沿って JSON を作成し、`sources[]` に出典を入れる
9. `node scripts/c-beta/check-cbeta-scenario-quality.mjs` を通す
10. immersive modeで `pnpm typecheck`、`pnpm lint`、`pnpm test:e2e` を通す
11. 没入モードを追加した場合は、序盤だけでなく中盤以降の会議・対話シーンまで E2E で進行確認する
12. `pnpm test`、`pnpm test:e2e`、`pnpm build` が通ったらmatrixと`docs/state.md`を更新する

Cβ 50シナリオ展開は、Cα安定化後にbatch単位で進める。50本を一括生成しない。
