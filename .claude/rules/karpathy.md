# Karpathy 4原則

すべてのコード生成・設計判断に適用する行動原則。

## 1. Think Before Coding（コード前に思考）

- 仮定（assumption）を作らず、明示された情報のみに基づいて判断する
- 不確かな点を隠さず、表面化させる（Surface Confusion）
- トレードオフを明示する。「Aを選んだ理由」と「Aを選ばなかった代償」を両方書く
- 答えがわからない時は推測せず、ユーザーに確認するか、調査する

## 2. Simplicity First（シンプルさ優先）

- 最小のコードで目的を達成する
- 投機的機能（speculative features）を実装しない
- 「将来必要になるかも」を理由に複雑性を導入しない
- 動く最小実装 → 必要に応じて拡張

## 3. Surgical Changes（外科的変更）

- 必要な箇所だけを変更する
- 既存のコードスタイル・命名規則・インデントに合わせる
- 関係ないファイルを「ついでに修正」しない
- リファクタリングは別タスクとして実施

## 4. Goal-Driven Execution（目的駆動実行）

- 命令でなく成功基準を提示する
- 検証ループ（テスト・lint・型チェック）で成功を確認する
- 「動いた気がする」で完了にしない
- ユーザーストーリーに対応する E2E テストで最終確認する

## WorkSim プロジェクト固有の適用

### Confusion を表面化する例

❌ 悪い例：「メール送信機能を実装したのでテストもしておきました」
✅ 良い例：「メール送信機能を実装しましたが、Supabase の Edge Function での実装が `vision.md` に明示されていないため、Next.js Route Handler で実装しました。これでよいか確認してください」

### Surgical Changes の例

シナリオJSONのスキーマを更新する時、既存の web-engineer.json は破壊しない。スキーマバージョニングを導入し、既存JSONはそのまま動作する状態を維持する。

### Goal-Driven の例

❌ 悪い例：「ログイン画面に Google ボタンを置く」
✅ 良い例：「US-2（Given/When/Then）が成立する。すなわち、未ログイン → ボタン押下 → Google承認 → プロフィール画面遷移、までエラーなく動作する」
